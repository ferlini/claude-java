你将扮演**项目维护者**的角色，基于本次会话到目前为止的所有工作，为当前分支的提交撰写一份清晰、面向团队审查者的 Pull Request 描述，并在用户确认后调用 `gh` CLI 创建 PR。

## 执行流程

### Step 1：收集上下文

并行执行以下命令了解当前分支与变更：

- `git status` —— 当前工作区状态
- `git log <base>..HEAD --oneline` —— 本分支相对基线分支的所有提交（基线分支默认 `main`，如不存在则用 `master`）
- `git diff <base>...HEAD` —— 本分支相对基线的完整差异
- 如果项目存在 `spec.md`，使用 Read 工具读取，提取核心用户故事用于"背景"章节

### Step 2：撰写 PR 描述

严格按照以下四段式结构生成 PR 描述（Markdown 格式）：

```markdown
## 背景（Context）

<简要说明这个 PR 要解决什么问题。如果有 spec.md，引用其中的核心用户故事；否则基于会话上下文与提交历史归纳。>

## 实现方案（Implementation）

<概述本次实现的关键改动：涉及的模块、设计取舍、关键数据结构或流程。用要点列表呈现。>

## 测试（Testing）

<说明本次开发遵循 TDD（测试先行），所有新增/变更代码均有单元测试覆盖。列出新增测试文件或覆盖的关键场景。>

## 如何手动验证（How to Verify）

<给出清晰的本地复现步骤，让审查者能够自行验证。包含：环境准备、命令、预期结果。使用编号列表。>
```

要求：
- **面向审查者**：避免内部黑话，让不参与开发的同事也能看懂
- **基于事实**：所有内容必须来自会话上下文、`git diff`、`spec.md`，不得编造
- **TDD 部分诚实**：如果本次工作并非纯 TDD（例如本次是配置/文档变更，无单元测试），如实说明，**不得**虚构测试

### Step 3：向用户展示并请求确认

将生成的标题与描述完整展示给用户，询问：

> 以上 PR 描述是否符合预期？确认后将执行 `gh pr create` 创建 PR。回复 **确认 / 修改 / 取消**。

### Step 4：创建 PR

收到用户**明确确认**后才执行：

1. 检查当前分支是否已推送到远程，未推送则执行 `git push -u origin <branch>`
2. 使用 HEREDOC 传递 body 调用 `gh pr create`：

   ```bash
   gh pr create --title "<title>" --body "$(cat <<'EOF'
   <Step 2 生成的四段式描述>
   EOF
   )"
   ```

3. 返回 PR URL 给用户

## 注意事项

- **不要**自动推送或创建 PR 而未经用户确认
- **不要**强制添加 `Co-Authored-By` 等签名行（除非用户要求）
- 如果当前分支就是 `main` / `master`，提示用户先创建 feature 分支再调用本命令
- 如果 `gh` 未登录，提示用户运行 `gh auth login` 后重试
